为什么有些软件的默认安装路径是C:\Users\用户名\AppData\Roaming? - root的回答 - 知乎 https://www.zhihu.com/question/548340950/answer/2637630471
summary the above, give me the key information in English

The key differences between the directories mentioned are:

  1. C:\Program Files - The installation directory for 64-bit software, accessible to all users. Admin permission is required to install software here. Storing program data in this directory poses a security risk.
  2. C:\Program Files(x86) - Similar to the above, but for 32-bit applications.
  3. C:\Users\Username\AppData\Roaming - Accessed through the %AppData% environment variable, stores user-specific data. This data is synced across computers in a domain network.
  4. C:\Users\Username\AppData\Local - Accessed through the %LocalAppData% environment variable, stores local user data that is not synced across computers in a domain network.
  5. C:\ProgramData - Contains data shared among all users. Use this directory for software data that should be consistent across users.
  6. C:\Users\Username - The user’s home directory, where personal data should be stored. Cross-platform software often generates hidden files with a dot prefix in this directory.
  7. C:\Users\Username\Documents - The “My Documents” folder, meant for user-generated data like documents, photos, music, videos, and archives. Some software may inappropriately store program data here.

The proper separation of software installation and data storage depends on the software developer and the installation package tool. It’s important to follow guidelines and avoid storing data in incorrect locations to ensure easier data backups and software upgrades.

“做到程序和数据分离，不要在程序安装目录下写数据，更不要把数据写到C:根目录Q下，简直是灾难。” why?

Separating program files and data is crucial for various reasons:

  1. Organization and clarity: Keeping program files and data separate makes it easier to manage and locate specific files when needed. Mixing data and program files can lead to confusion and difficulty in finding essential files.

  2. Security: Storing data in program directories, especially those that require administrator permissions, can expose the data to security risks. Malicious software may take advantage of the elevated permissions and compromise the data.

  3. Data integrity: Separating program files and data ensures that software updates or uninstallation processes do not accidentally delete or modify essential data. This separation helps maintain the integrity and safety of your data.

  4. Backup and migration: Keeping data separate from program files simplifies the backup process and makes it easier to transfer data between different computers or devices. This separation allows users to back up essential data without including unnecessary program files, making data management more efficient.

  5. User management: In multi-user environments, different users might have different access levels and data requirements. Separating program files and data allows for more granular control over access permissions and ensures that user-specific data is kept separate and private.

Storing data directly in the C:\ root directory is particularly problematic because it makes the file system disorganized, difficult to manage, and potentially exposes your data to security risks. Moreover, it goes against best practices for organizing files and directories on a computer.

why x86 is for 32-bit app

The term “x86” is used to refer to 32-bit applications because it originates from the Intel 8086 processor family. The 8086 was a 16-bit processor introduced by Intel in 1978. As new processors were developed in the same line, they maintained backward compatibility with the 8086 instruction set, which is why they were named with an “86” suffix, like the 80186, 80286, 80386, and 80486.

When the 80386 was introduced, it brought support for 32-bit processing while still retaining compatibility with the older 16-bit x86 instruction set. This extended architecture was called IA-32 (Intel Architecture, 32-bit). As time went on and newer processors were released, the x86 nomenclature persisted and became synonymous with 32-bit architecture, even though it originated from a 16-bit processor family.

In contrast, the x64 architecture, also known as x86-64 or AMD64, was introduced by AMD to extend the x86 architecture to 64 bits, while maintaining backward compatibility with 32-bit applications. So, the “x86” term is used for 32-bit applications to distinguish them from the newer 64-bit (x64) applications.

so amd invent 64-bit processer?

AMD did not invent the 64-bit processor, but they were the first to extend the x86 architecture to 64 bits. The first 64-bit processor architecture was actually developed by MIPS Technologies, with the MIPS R4000, released in 1991.

However, when it comes to the x86 architecture commonly used in personal computers, AMD played a significant role. They introduced the x86-64 (AMD64) architecture, an extension of the existing 32-bit x86 architecture that added 64-bit capabilities while maintaining backward compatibility with 32-bit applications.

Intel, the original creator of the x86 architecture, later adopted a similar approach with their own 64-bit extension called Intel 64 (formerly known as EM64T). Intel 64 and AMD64 are largely compatible, and together they form the x86-64 architecture that is widely used in modern personal computers.

AMD (Advanced Micro Devices):

AMD is an American multinational semiconductor company founded in 1969 by Jerry Sanders and headquartered in Santa Clara, California. The company specializes in developing and manufacturing computer processors, graphics processing units (GPUs), and related technologies for both consumer and enterprise markets.

AMD is best known for its line of x86-64 CPUs, such as the Athlon, Phenom, Ryzen, and EPYC processors. They have been a key competitor to Intel in the CPU market. AMD also acquired ATI Technologies in 2006, which enabled them to enter the GPU market and compete against NVIDIA with their Radeon graphics cards.

Intel (Integrated Electronics Corporation):

Intel is an American multinational technology company founded in 1968 by Robert Noyce and Gordon Moore, headquartered in Santa Clara, California. It is one of the world’s largest semiconductor chip manufacturers and is best known for its line of x86 microprocessors, which are found in a majority of personal computers.

Intel’s product lineup includes a wide range of processors, such as the Core i3, i5, i7, and i9 series, as well as the Xeon processors used in servers and workstations. In addition to CPUs, Intel also manufactures chipsets, motherboards, solid-state drives (SSDs), network interface cards, and other computer components.

Both AMD and Intel are industry leaders in the development of computer processors and have played significant roles in shaping the technology landscape over the past few decades. While Intel has traditionally dominated the CPU market, AMD has made significant strides in recent years, offering competitive products in terms of performance and price.

what is wiki

A wiki is a collaborative online platform that allows users to create, edit, and maintain interlinked web pages. It serves as a knowledge base where information can be collected, organized, and shared among people. The most famous example of a wiki is Wikipedia, a free, user-generated encyclopedia that contains articles on a wide range of topics.

The term “wiki” originates from the Hawaiian word “wiki-wiki,” which means “quick” or “fast.” This is because wikis are designed to enable rapid and easy content creation, editing, and organization by multiple users.

Wikis often have a simple markup language or a rich-text editor, which allows users to format text, add links, and insert multimedia elements without having to learn complex programming languages. Access to wikis can be open to the public or restricted to specific user groups, depending on the purpose and intended audience.

in github

GitHub is a web-based platform primarily used for version control and collaboration in software development. While it’s not a wiki platform in the traditional sense, it does offer a feature called GitHub Wiki, which allows you to create and maintain documentation or other project-related information alongside your code repository.

To create and use a wiki on GitHub, follow these steps:

  1. Navigate to your GitHub repository.
  2. Click on the “Wiki” tab near the top of the page, next to “Issues” and “Pull requests.”
  3. If no wiki has been created yet, click on the “Create the first page” button.
  4. Enter a title for your wiki page and add the content using the Markdown editor.
  5. Click the “Save page” button to save your changes.

Once you’ve created the initial wiki page, you can add more pages, edit existing pages, and manage the wiki as needed. GitHub Wiki supports Markdown syntax, which allows you to format text, create tables, add links, and insert images or other multimedia elements.

Keep in mind that by default, the GitHub Wiki is publicly accessible, and anyone with a GitHub account can edit it. However, you can change the settings to restrict editing to collaborators only by navigating to your repository settings and selecting the “Restrict editing to collaborators only” checkbox under the “Wikis” section.

how can i write effective git message

Writing effective Git commit messages is crucial for maintaining a clean and understandable project history. Good commit messages help you and your collaborators quickly understand the changes made in each commit. Here are some guidelines to write effective Git commit messages:

  1. Write a concise and descriptive subject line: The first line of the commit message should be a short summary (up to 50 characters) of the changes made. Begin with a capital letter and use imperative mood (e.g., “Add”, “Fix”, “Update”).

  2. Separate the subject from the body with a blank line: If you need to provide more details about the commit, leave a blank line after the subject line and then write the body. This separation makes the message easier to read.

  3. Wrap the body at 72 characters: When writing a longer commit message with a body, limit each line to 72 characters. This ensures that the message is properly formatted and easy to read in various Git tools.

  4. Use the body to explain what and why: The body should explain what changes were made and the reasons for those changes. Focus on the context and the motivation behind the changes, rather than describing the exact lines of code that were altered.

  5. Use bullet points or lists: If you need to describe multiple changes, use bullet points or lists to make the commit message more readable.

  6. Use proper grammar and punctuation: Ensure your commit messages are well-written, with proper grammar, punctuation, and spelling. This makes the messages easier to understand and looks more professional.

must it start with fix/update/add?

No, your Git commit message doesn’t have to start with “fix”, “update”, or “add”. These words are just common examples that follow the convention of using the imperative mood in commit messages. The main idea is to use a verb in the imperative form that briefly and clearly describes the action taken in the commit.

Some other examples of verbs you can use are:

  • Remove
  • Change
  • Implement
  • Refactor
  • Merge
  • Create
  • Optimize
  • Reformat

Choose the verb that best represents the change made in the commit. The most important aspect is to make the commit message concise, informative, and easy to understand.
